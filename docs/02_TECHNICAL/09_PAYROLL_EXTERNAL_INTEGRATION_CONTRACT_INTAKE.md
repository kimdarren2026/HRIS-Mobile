# Phase 34A — Payroll External Integration Contract Intake

## Purpose

- HRIS Mobile is not the payroll calculation/payment engine.
- HRIS Mobile is the source of truth for employee master data and attendance.
- Salary calculation/payment will be handled by a separate external payroll system.
- HRIS Mobile may later receive payroll/payslip results from the external payroll system.
- This phase only documents questions and contract requirements before coding.

## Current Architecture

- HRIS owns employee data.
- HRIS owns attendance data.
- External payroll owns salary calculation, payroll approval/payment, and final payroll result.
- HRIS may display payslip results later after a contract/API/file format is agreed.
- No internal payroll workflow should be reintroduced.

## Questions for the Payroll External Team

- Who is the payroll system owner/PIC?
- Will integration use API, CSV, Excel, SFTP, manual upload, or another method?
- Will payroll pull data from HRIS or will HRIS push data to payroll?
- What employee fields are required?
- What attendance fields are required?
- What leave/absence fields are required?
- What period/date format is required?
- What unique identifier will match employees between systems?
- What payroll result fields will be returned to HRIS?
- Will payslip files be generated by payroll system or HRIS?
- Will HRIS store PDF payslips or only display structured data?
- What statuses are needed, such as draft, calculated, approved, paid, cancelled, failed?
- What authentication method is required?
- What IP/domain allowlist is required?
- What audit logs are required?
- What error handling and retry rules are required?
- How often should sync happen?
- What data retention rules apply?

## Draft Data Contract Sections

### Employee Master Data Export Candidate Fields

- Employee unique identifier
- Employee number/NIK or external employee ID
- Full name
- Employment status
- Department
- Position
- Join date
- Work location or office location
- Email or contact field, if required by payroll
- Last updated timestamp

### Attendance Export Candidate Fields

- Employee unique identifier
- Attendance date
- Check-in time
- Check-out time
- Attendance status
- Outside-radius flag or review status
- Approval status and approver metadata, if required
- Late/absence indicators, if agreed
- Last updated timestamp

### Leave/Absence Export Candidate Fields

- Employee unique identifier
- Leave or absence type
- Start date
- End date
- Total days
- Approval status
- Approval timestamp
- Approved by
- Last updated timestamp

### Payroll Result Import Candidate Fields

- Payroll period identifier
- Employee unique identifier
- Gross pay, if allowed to be stored in HRIS
- Deductions, if allowed to be stored in HRIS
- Net pay, if allowed to be stored in HRIS
- Payroll status
- Payment status
- Payment date
- External payroll reference
- Generated timestamp

### Payslip Result Candidate Fields

- Payroll period identifier
- Employee unique identifier
- Payslip reference ID
- Payslip file path or download URL, if payroll owns file generation
- Structured payslip payload, if HRIS displays structured data
- Published timestamp
- Visibility status

### Error Response Candidate Fields

- External request ID
- Error code
- Error message
- Field name or record identifier
- Retryable flag
- Occurred timestamp

### Audit Log Candidate Fields

- Integration action
- Direction: HRIS to payroll or payroll to HRIS
- External request ID
- Record count
- Result status
- Error summary without sensitive payloads
- Actor or system identity
- Timestamp

## Security Requirements

- Do not expose bank/account/salary data publicly.
- Use authenticated integration only.
- Use least privilege.
- Log integration activity without leaking sensitive payloads.
- Avoid storing secrets in code or documentation.
- Production credentials must be managed outside Git.
- Any payslip access must be scoped to the owning employee and authorized HR/super_admin users.

## Decisions Not Made Yet

- API vs file-based integration.
- Exact field names.
- Exact authentication.
- Sync schedule.
- Payslip storage method.
- Payroll status lifecycle.
- Error/retry behavior.

## Acceptance Criteria

- Intake document exists.
- No payroll code implemented.
- No application code changed.
- No `.env` changed.
- No deploy or migration run.
- Integration remains blocked until external payroll format/API is confirmed.
